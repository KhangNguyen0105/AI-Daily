# Phase 8 — UI Design Contract

> Visual and interaction contract for the Admin Operations phase. Documents the existing design decisions and patterns used in the admin UI. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (custom Tailwind components) |
| Preset | not applicable |

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (normal) | 1.5 |
| Label | 12px (`text-xs`) | 600 (semibold), uppercase, tracking-wide | 1.4 |
| Heading (page) | 24px (`text-2xl`) | 700 (bold) | 1.3 |
| Heading (section) | 18px (`text-lg`) | 600 (semibold) | 1.3 |
| Display | 32px (`text-2xl` in SummaryCard value) | 700 (bold) | 1.2 |

**Typography rules:**
- All body text uses `text-sm` (14px) — never 16px in admin UI
- Table headers use `text-xs font-semibold uppercase tracking-wide text-gray-500`
- Page titles use `text-2xl font-bold text-gray-900`
- Section headings use `text-lg font-semibold text-gray-900`
- Labels use `text-sm font-medium text-gray-700`

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` (white) | Page background, cards, sidebar, table backgrounds |
| Secondary (30%) | `#F9FAFB` (`bg-gray-50`) | Table header rows, expanded row details, login page background |
| Accent (10%) | `#2563EB` (`bg-blue-600`) | Primary buttons, active nav links, focus rings, links |
| Destructive | `#DC2626` (`bg-red-600`) | Destructive confirm buttons, error text, rollback links |

Accent reserved for:
- Primary action buttons (`bg-blue-600 hover:bg-blue-700`)
- Active sidebar links (`text-blue-600`, `bg-blue-50`)
- Active tab indicators (`border-blue-600`, `text-blue-600`)
- Focus rings (`focus:ring-blue-600` or `focus:ring-blue-500`)
- Inline links (`text-blue-600 hover:text-blue-800`)

**Status colors:**
| Status | Background | Text |
|--------|-----------|------|
| Healthy / Completed / Active | `bg-green-100` | `text-green-800` (badge) or `text-green-600` (icon) |
| Error / Failed | `bg-red-50` or `bg-red-100` | `text-red-600` (text) or `text-red-800` (badge) |
| Running / Warning | `bg-yellow-100` | `text-yellow-600` |
| Inactive / Draft / Unknown | `bg-gray-100` | `text-gray-600` |

**Toggle switch colors:**
- On state: `bg-blue-600`
- Off state: `bg-gray-300`
- Thumb: `bg-white` with `shadow`

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (save) | "Save Changes" |
| Primary CTA (sign in) | "Sign In" |
| Empty state heading (articles) | "No articles yet" |
| Empty state body (articles) | "Articles will appear here once the daily content engine publishes its first digest." |
| Empty state heading (pipeline runs) | "No pipeline runs recorded" |
| Empty state body (pipeline runs) | "Pipeline runs will appear here after the first daily collection completes." |
| Empty state heading (errors) | "No errors recorded" |
| Empty state body (errors) | "All recent pipeline runs completed without errors." |
| Empty state heading (versions) | "No previous versions" |
| Empty state body (versions) | "This article has not been edited yet. Save changes to create a version history." |
| Empty state heading (sources) | "No sources found" |
| Empty state body (sources) | "Try adjusting your filters, or check that providers have been configured." |
| Error state (login) | "Invalid password. Please try again." |
| Error state (generic) | "{Action} failed. {What went wrong}. {What to try}." |
| Loading state | "Loading {noun} data..." |
| Destructive: re-crawl | "Re-crawl Provider" / "This will queue a new crawl job for {provider}. The current data will remain until the crawl completes. Continue?" / "Yes, Re-crawl" |
| Destructive: regenerate | "Regenerate Article" / "This will regenerate the article for {date}. The current version will be saved in history. Continue?" / "Yes, Regenerate" |
| Destructive: rollback | "Rollback Article" / "This will replace the current article content with a previous version. The current version will be saved in history. Continue?" / "Yes, Rollback" |
| Destructive: full pipeline | "Run Full Pipeline" / "This will start a full pipeline run, crawling all providers and generating a new digest. Continue?" / "Yes, Run Pipeline" |

**Copy rules:**
- All toast messages end with a period
- Error toasts include what went wrong AND what to try next
- Success toasts are brief confirmation statements
- Confirmation dialog messages always end with "Continue?"
- Confirmation button text starts with "Yes, " prefix

### Interactive Components

**ConfirmDialog** (`app/components/admin/ConfirmDialog.tsx`)
- Modal overlay: `fixed inset-0 bg-black/50 z-50`
- Card: `bg-white rounded-lg p-6 max-w-md shadow-xl`
- Focus trap: auto-focus confirm button on open
- Escape key to dismiss
- Backdrop click to dismiss
- Two variants: `default` (blue confirm) and `danger` (red confirm)

**Toast** (`app/components/admin/Toast.tsx`)
- Hook: `useToast()` returns `{ toasts, addToast, removeToast }`

- Auto-dismiss: 5 seconds
- Left border: 4px, green (success) or red (error)
- Dismiss button: X icon

**AutoPublishToggle** (`app/components/admin/AutoPublishToggle.tsx`)
- `role="switch"` button, 24x44px touch target
- Optimistic UI: toggle immediately, revert on API failure
- Disabled state: `opacity-50`

## Interaction Patterns

### Expandable Table Rows
- Click row or anywhere in the row to toggle
- Only one row expanded at a time (accordion pattern)
- Expanded content: `bg-gray-50/80 shadow-inner` with full-width colSpan
- Transition: CSS transition on background color

### Optimistic Updates (Toggle Switches)
- Toggle fires immediately (UI updates before API response)
- On API failure: revert to previous state
- No loading spinner on toggle itself (only `opacity-50` during pending)

### Confirmation Flow
1. User clicks action button
2. ConfirmDialog opens with action-specific title, message, and confirm label
3. User confirms -> API call fires, button shows loading state
4. Success -> toast notification, dialog closes, data refreshes
5. Error -> toast notification with error details, dialog closes

### Data Refresh
- Pipeline page: SSE (`EventSource`) for real-time run updates + initial `fetch` load
- Sources page: manual refresh via re-fetch after toggle
- Article edit: manual refresh via re-fetch after save/rollback
- No SWR implemented — uses direct `fetch` with manual state management
