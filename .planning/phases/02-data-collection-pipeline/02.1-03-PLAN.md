**Source Confidence (per D-07, 25% weight):**
- Tier 1 (official pricing page): 85-90
- Tier 2 (official blog/changelog): 70-80
- Tier 3 (aggregators): 50-65
- Adjusted by: is_official_source boolean, source_url_confidence from evidence

**Extraction Confidence (25% weight):**
- hasAllFields (model_name, input, output, context): 85-90
- hasCoreFields (model_name, input, output): 70-75
- missingContext: 50-60
- Missing core field: 20-30
- Adjusted by: LLM confidence score if available

**Normalization Confidence (20% weight, from D-05):**
- high (direct per-1M tokens): 85-90
- medium (arithmetic conversion from per-1K to per-1M): 70-75
- low (non-token plan): 40-50
- unknown (contact sales): 10-20

**Verification Confidence (20% weight):**
- Pass B matched (0.1% tolerance): 85-90
- Pass B matched with warning (edge case detected): 70-75
- Pass B disagreement detected: 20-30
- No second pass: 50-60

**Freshness Confidence (10% weight, computed by freshness tracker):**
- Fresh (<24h): 90
- Recent (24-72h): 70
- Aging (3-7d): 40
- Stale (>7d): 20
- Very stale (>14d): 10

**Overall Confidence (weighted combination):**
```
overall = (
  source_confidence * 0.25 +
  extraction_confidence * 0.25 +
  normalization_confidence * 0.20 +
  verification_confidence * 0.20 +
  freshness_confidence * 0.10
)
```

**Critical Rules (per D-07):**
- If input_price_confidence < 40 OR output_price_confidence < 40 → overall cannot exceed 54 (Medium max)
- If source is unofficial → overall cannot exceed 74 (High max)
- If verification disagreement detected → record quarantined (<30)
- If edge_case detected (free tier, monthly plan, etc.): add note "verified_with_warning"

**Per-Field Confidence:**
- For each critical field (model_name, input_price, output_price, context_window, currency, pricing_unit, free_tier, source_url):
  - Based on field-specific evidence quality + extraction completeness + verification pass on that field
  - If field missing from extraction: field_confidence = 10-20
  - If field present but low-confidence extraction: field_confidence = 40-50
  - If field verified and freshness good: field_confidence = 85-95

**Human Override (per D-07):**

```typescript
interface HumanReviewOverride {

```

When human override applied:
- confidence_override replaces overall_confidence in UI
- Keep machine_score in DB for audit trail
- Record human_review_status in extractions table
- Per D-07: Manual review sits on top, does not erase machine scores

**Label Mapping:**
- overall_confidence >= 90: Verified
- overall_confidence 75-89: High
- overall_confidence 55-74: Medium
- overall_confidence 30-54: Low
- overall_confidence < 30 or quarantined: Quarantined

calculateMultiDimensionalConfidence must be called after:
1. extraction complete (source + extraction + normalization confidence set)
2. verification complete (verification + freshness confidence available)
3. Optional: human override applied

Per D-07 decision: Dimensions allow tooltip to explain "Source high, Extraction medium, Freshness low = Medium confidence overall".

**SLA Tiers (per D-03):**

| Tier | Providers | Crawl Frequency | Freshness SLA | Badge Threshold | Warning Threshold |
|------|-----------|-----------------|----------------|-----------------|-------------------|
| Tier 1 | OpenAI, Anthropic, Google, xAI, DeepSeek, Mistral | Every 2-4 hours | <24 hours | Stale >24h | >14 days |
| Tier 2 | Cohere, Perplexity, Groq, Together, Fireworks, OpenRouter | Every 6-12 hours | <48 hours | Stale >48h | >14 days |
| Tier 3+ | All others | Daily | <7 days | Stale >7d | >14 days |

**Automatic Actions on SLA Breach (per D-03):**
- Trigger priority recrawl (enqueue highest priority in BullMQ)
- Generate admin alert (log in admin dashboard)
- Log freshness incident (in pipeline_runs stats)
- After repeated failures (3+ consecutive breaches): mark provider as degraded, surface warning

**Freshness Badge Display (per D-03, in API response):**
- Fresh (<24h): green badge
- Recent (24-72h): blue badge
- Aging (3-7d): amber badge
- Stale (>7d): red badge
- Very stale (>14d): red badge + warning banner
- Excluded (>30d): excluded from rankings by default, marked "Data unavailable"

**Implementation flow:**
1. Extract worker creates extraction with extracted_at = NOW()
2. Score worker calls computeFreshnessMetadata(extractionId, sourceTier) → sets last_verified_at, freshness_status
3. Score worker calls checkSLABreach(sourceId, providerName, sourceTier) → if breached, trigger alert + recrawl
4. Score worker applies computeFreshnessConfidence(dataAgeMinutes) to confidence score
5. Pricing API calls computeFreshnessMetadata for each extraction → returns badge status

Per D-03 decision: Freshness tracked at multiple levels (global + per-provider + per-model). Confidence scoring penalizes stale data heavily.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Confidence Score → UI Display | Machine score is informational; human override is source of truth if present |
| Freshness Data → Confidence Calculation | Stale data age directly reduces freshness dimension |
| SLA Enforcement → Admin Alert | Breaches surface in admin dashboard but don't auto-exclude from UI (manual action required) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-11 | Tampering | Confidence dimension weights | mitigate | Weights (0.25, 0.25, 0.20, 0.20, 0.10) hardcoded in code; not user-configurable in Phase 2.1 |
| T-02-12 | Information Disclosure | Confidence breakdown reveals LLM extraction quality | accept | Breakdown is intended for transparency; no PII in fields |
| T-02-13 | Denial of Service | Freshness check on every pricing request | mitigate | computeFreshnessMetadata uses indexed queries (source_id, last_verified_at); caches computation for 1 hour |
| T-02-14 | Spoofing | Human override without audit trail | mitigate | reviewed_by, reviewed_at, review_notes stored in DB; machine_score preserved for comparison |
