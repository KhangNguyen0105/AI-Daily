## Task 2: CanonicalRegistry Class

### What Was Built

**File:** `src/lib/canonical-registry.ts` (327 lines)

#### Class Methods (6)

1. **`register(providerName, providerModelId, extraction, sourceTier): Promise<string>`**
   - Creates or retrieves canonical model by provider ID priority
   - Returns existing ID if provider model ID matches (highest confidence per D-04)
   - Appends new provider model IDs to existing api_model_ids[]
   - Records "created" audit event
   - Idempotent: safe to re-run on same input

2. **`resolve(providerName, extractedName, providerModelId?): Promise<{canonical_id, canonical_name} | null>`**
   - **Priority order (per D-04):**
     1. Provider model ID exact match (highest)
     2. Canonical alias exact match (case-insensitive)
     3. Normalized name match
     4. Return null if none match
   - Returns null for no match

3. **`addAlias(canonical_id, alias): Promise<void>`**
   - Appends alias to aliases[] array
   - Records "aliased" audit event
   - Skips if alias already present (idempotent)

4. **`recordRename(canonical_id, old_name, new_name, reason): Promise<void>`**
   - Updates canonical_name to new_name
   - Appends old_name to aliases[] if not present
   - Records rename in lineage.renames[] with timestamp and reason
   - Records "renamed" audit event

5. **`getLineage(canonical_id): Promise<Array<{type, timestamp, ...}>>`**
   - Returns array with:
     - model_created event (from first_seen + lineage)
     - All audit events (created, renamed, aliased, deprecated, replaced, quarantined) in chronological order
   - Full rename chain visibility

6. **`markDeprecated(canonical_id, replacedBy?): Promise<void>`**
   - Sets status to "deprecated" or "replaced" (if replacedBy provided)
   - Updates lineage.deprecatedAt and lineage.replacedBy
   - Records appropriate audit event

#### Supporting Methods

- `resolveByProviderId(providerName, providerModelId)`: Private, checks api_model_ids[] for provider ID match
- `normalizeName(name)`: Lowercase, trim, remove punctuation, collapse dashes

### Key Decisions Applied

- **Per D-04:** Provider model ID is highest-priority identity signal (checked first)
- **Per D-04:** Canonical_id immutable; lineage tracks all transformations
- **Per D-04:** Hybrid matching: exact first (via provider ID), then aliases, then fuzzy (normalized name)
- Idempotent methods: safe for replay without duplicates

## Task 3: Deduplication Logic

### What Was Built

**File:** `src/lib/deduplication.ts` (328 lines)

#### Core Functions (3)

1. **`detectDuplicates(extraction, providerName, registry): Promise<DuplicateDetectionScore>`**
   - Queries all canonical models for provider
   - Calculates match score for each candidate
   - Sorts by score descending
   - Returns:
     - candidates[] with scores and signals
     - recommendation: 'auto_link' | 'review_queue' | 'new_model'

2. **`calculateMatchScore(extraction, candidate): Promise<{score, signals}>`**
   - Weighted matching algorithm with 5 signals:
     1. **Provider ID exact match:** 100% (binary)
     2. **Alias exact match (case-insensitive):** 90%
     3. **Normalized name match:** 75%
     4. **Family + context window match (within 10%):** 60%
     5. **Pricing similarity (within 5% on both input/output):** 40%
   - Final score = highest non-zero signal (hierarchical, not additive)
   - Returns score (0-100) + breakdown of all signals

3. **`generateReviewQueue(extraction, extractionId, detectionScore): Promise<ReviewQueueItem>`**
   - Creates admin-facing review item for medium-confidence candidates
   - Includes:
     - extraction_id, extracted_model_name
     - candidates[] with match_score, match_reason, recommendation_strength
     - status: 'pending', created_at

#### Recommendation Logic

| Score | Recommendation | Use Case |
|-------|----------------|----------|
| > 85% | `auto_link` | High confidence, auto-link to canonical model |
| 55-84% | `review_queue` | Medium confidence, route to admin dashboard |
| < 55% | `new_model` | Low confidence, create new canonical model |

#### Supporting Types

```typescript
interface MatchSignals {
  providerIdMatch?: number;
  aliasMatch?: number;
  normalizedNameMatch?: number;
  familyAndContextMatch?: number;
  pricingSimilarity?: number;
}

interface DuplicateCandidate {
  canonical_id: string;
  canonical_name: string;
  matchScore: number;
  signals: MatchSignals;
  confidence: 'high' | 'medium' | 'low';
}

interface DuplicateDetectionScore {
  modelName: string;
  candidates: DuplicateCandidate[];
  recommendation: 'auto_link' | 'review_queue' | 'new_model';
}
```

### Key Decisions Applied

- **Per D-04:** Fuzzy matching candidates-only, never auto-merge below 85%
- **Per D-04:** Provider ID match is 100% (binary, not scored with others)
- Scoring hierarchy: provider ID > alias > name > family > pricing
- Pricing similarity weak signal (40%, used only as tiebreaker)

## Decisions Made

| Decision | Rationale |
|----------|-----------|
| Provider-ID-first resolution | Highest trust signal per D-04; API model IDs more stable than names |
| Hierarchical signal scoring | Easier to reason about (one clear winner); prevents false tiebreaker scenarios |
| 85% threshold for auto_link | Conservative; reduces risk of incorrect auto-merges across providers |
| Append-only pricing_history | Enables audit trail, prevents data loss from overwrites (per D-05) |
| UUID for canonical_id | Immutable identity; no business meaning; prevents accidental collisions |
| Lineage JSONB | Flexible schema for rename chains, replacements, version history |
