# Phase 2.1 Integration & Safety Plan

## Executive Summary

Phase 2.1 introduces significant schema changes (canonical registry, deduplication, freshness tracking) while Phase 3 (Pricing Table), Phase 4 (Cost Calculator), and Phase 5 (Model Detail Pages) are already live or in progress.

**Strategy:** Dual-write transition with feature flags and rollback capability. No breaking changes to existing APIs or UI.

---

## 1. Existing Pricing Data Migration Strategy

### Current Data State

**Phase 2 produced:**
- ~5,000-10,000 extraction records (12 providers × ~500 models each)
- No canonical model deduplication (models may be listed under multiple names)
- No pricing history (each crawl overwrites previous prices in `extractions` table)
- Single-dimension confidence (verified/likely/low-confidence)
- No freshness tracking per model

**API Consumers:**
- Phase 3 (Pricing Table): `/api/pricing/table` → queries `extractions` table directly
- Phase 4 (Cost Calculator): `/api/pricing/scenarios` → queries `extractions` table
- Phase 5 (Model Detail): `/api/models/[slug]` → queries `extractions` + manual slug mapping
- Admin Dashboard (Phase 8): `/api/admin/extractions` → raw extraction management

### Migration Plan (Wave 1 Execution)

**Phase 1A: Add new schema, run migrations (zero downtime)**
```sql
-- New tables (canonical_models, pricing_history, model_status_audit)
-- are added WITHOUT dropping any existing tables
-- Existing extractions table unchanged initially
-- Run: npx drizzle-kit push (doesn't drop, only adds)
```

**Phase 1B: Data backfill (async batch job, during off-hours)**
```
Timing: During daily pipeline gap (not during scheduled collection/generate cycles)

For each unique model currently in extractions:
  1. Call CanonicalRegistry.register(provider, modelId, extraction, tier)
  2. Creates canonical_models record with status='active'
  3. Updates extraction.canonical_model_id = canonical_id
  4. Copies pricing to pricing_history (append-only record)

Duration: ~30 minutes for 10k extractions (50-100 records/sec batched)
Status: Logged with progress updates
Rollback: Can re-run safely (idempotent — checks for existing canonical_id)
```

**Phase 1C: Verify backfill integrity**
```
After backfill:
  - Count canonical_models records
  - Verify all extractions.canonical_model_id IS NOT NULL
  - Check pricing_history has same price records as extractions
  - Spot-check 20-30 records manually for accuracy
```

### Backward Compatibility (Phase 1-3)

**Extraction records remain unchanged:**
- Existing `extractions` table continues to work
- All columns preserved (input_price_per_1m, output_price_per_1m, model_name, etc.)
- New columns (canonical_model_id, freshness_status, raw_price_text) added as nullable during transition

**Pricing Table (Phase 3) continues uninterrupted:**
- Current query: `SELECT * FROM extractions WHERE confidence != 'low' ORDER BY provider`
- This query still works (no breaking schema changes)
- Canonical model ID is available as optional join target but not required

**Cost Calculator (Phase 4) continues uninterrupted:**
- Current calculation uses extractions.input/output prices
- Pricing model type default: NULL → treated as 'token_usage' (backward compatible)

### Database Query Backward Compatibility (View Layer)

Create database views for Phase 3 queries to isolate changes:

```sql
-- View: current_extractions_for_pricing_table
-- This is what Phase 3 queries
CREATE VIEW current_pricing_view AS
SELECT
  e.id as extraction_id,
  COALESCE(e.canonical_model_id, e.id::text) as model_id,
  e.model_name,
  e.provider,
  e.input_price_per_1m,
  e.output_price_per_1m,
  e.confidence,
  COALESCE(e.freshness_status, 'fresh') as freshness_status,
  e.updated_at as last_updated
FROM extractions e
WHERE e.confidence != 'low_confidence'
ORDER BY e.provider, e.model_name;

-- Phase 3 query continues to work unchanged:
SELECT * FROM current_pricing_view;
```

This view abstracts the schema; Phase 3 doesn't need to know about canonical registry.

## 3. Rollback Plan If Canonical Registry Migration Fails

### Failure Scenarios & Responses

**Scenario A: Backfill integrity check fails (e.g., orphan extractions detected)**

```
Symptom: Backfill process reports extraction_ids without matching canonical_model_id

Response (immediate, <15 min):
  1. Set CANONICAL_REGISTRY_ENABLED = false (via env var / config)
  2. Queries automatically fall back to extractions table view
  3. All APIs continue responding (view layer masks schema)
  4. Investigate orphan records
  5. Manual cleanup or re-run backfill
  6. Resume Phase 2.1

Rollback time: <2 minutes (single config change)
Data impact: None (canonical tables unmodified; extractions intact)
```

**Scenario B: Deduplication detection is creating false positives**

```
Symptom: detectDuplicates function incorrectly merges distinct models
Example: "gpt-4o" matched with "gpt-4" at 78% (false positive)

Response (if detected during Wave 1 testing):
  1. Disable dedup worker: DEDUP_WORKER_ENABLED = false
  2. New extractions skip dedup processing (canonical_model_id = null)
  3. Query falls back to extraction_id
  4. No public UI impact (view layer handles null canonical_id)
  5. Fix matching algorithm, add test case
  6. Re-run backfill with fixed logic
  7. Enable dedup worker

Rollback time: <5 minutes
Data impact: None (no data deleted; new extractions just skip dedup)
```

**Scenario C: Canonical registry migration locks table, blocks extraction worker**

```
Symptom: Extract worker hangs waiting for canonical_models table lock

Response (if detected during backfill):
  1. Cancel ongoing backfill process
  2. Extractions table remains writeable (no locks)
  3. Extract → Dedup chain pauses (dedupQueue remains empty)
  4. Next scheduled crawl may be delayed <1 hour
  5. Scale down backfill job, retry with smaller batches
  6. Or defer backfill until off-hours window

Mitigation: Use non-blocking PostgreSQL migration tool (Alembic / Flyway) + table-level locks
```

**Scenario D: Complete rollback to pre-Phase-2.1 state**

```
Nuclear option (only if Wave 1 corrupts data):

  1. pg_restore from pre-Phase-2.1 backup
  2. Revert code changes: git checkout HEAD~N src/
  3. Restart app container
  4. Existing Phase 3-8 continue from backup state

Estimated time: 30 minutes
Data loss: ~24 hours of new extractions (if backfill was overnight)
Frequency: Should never happen if integration tests pass first
```

### Rollback Checklist (Before Wave 1 Execution)

- [ ] Pre-Phase-2.1 database backup created and tested
- [ ] Backup restoration procedure documented and rehearsed
- [ ] CANONICAL_REGISTRY_ENABLED / DEDUP_WORKER_ENABLED flags added to env config
- [ ] View layer (current_pricing_view) created and tested
- [ ] Phase 3 API routes tested against view layer
- [ ] Mock data backfill script created and tested on staging
- [ ] Rollback procedures documented and accessible to on-call engineer
- [ ] Logging/alerting set up for migration failures

## 4. Transition Strategy: Feature Flags vs. Dual-Write

### Approach: Feature Flags + Gradual Opt-In (NOT dual-write)

**Why NOT dual-write:**
- Dual-write (writing to both old `extractions` and new `canonical_models` simultaneously) creates consistency problems
- If `canonical_models` write fails but `extractions` succeeds → data drift
- If `extractions` write fails but `canonical_models` succeeds → orphan records
- Dual-write is harder to rollback; canonical registry becomes entangled

**Why feature flags:**
- Clean separation: enable by feature, not by write path
- Single source of truth during transition
- Easy to roll back (one flag change)
- Easier to test (enable flags in test environment only)

### Feature Flag Strategy

```ts
// src/config/phase-2-1.ts
export const PHASE_2_1_FLAGS = {
  // Wave 1 execution
  CANONICAL_REGISTRY_ENABLED: process.env.FEATURE_CANONICAL_REGISTRY === 'true', // false during backfill
  DEDUP_WORKER_ENABLED: process.env.FEATURE_DEDUP_WORKER === 'true', // false during backfill
  
  // Wave 2 execution (after backfill stabilizes for 24h)
  EVIDENCE_ANCHORING_REQUIRED: process.env.FEATURE_EVIDENCE_ANCHORING === 'true', // false during backfill
  
  // Wave 3 execution (after Wave 2 is stable)
  FRESHNESS_SLA_ENFORCEMENT: process.env.FEATURE_FRESHNESS_SLA === 'true', // false during Wave 2
  MULTI_DIM_CONFIDENCE_ENABLED: process.env.FEATURE_MULTI_DIM_CONFIDENCE === 'true', // false during Wave 2
};
```

### Query Path During Transition

**Wave 1 Execution (flags disabled):**
```
Phase 3 requests: GET /api/pricing/table
→ Query layer: SELECT * FROM current_pricing_view
  WHERE canonical_registry_enabled = false
  → Fallback: SELECT FROM extractions (unchanged)
→ Response: Original format + optional new fields (ignored by Phase 3)
```

**Post-Wave-1 (flags enabled):**
```
Phase 3 requests: GET /api/pricing/table
→ Query layer: SELECT * FROM current_pricing_view
  WHERE canonical_registry_enabled = true
  → Include canonical_model_id, freshness_status (JOIN to canonical_models)
→ Response: Enhanced format with new fields
→ Phase 3 ignores new fields (backward compatible)
```

### Dedup Worker Integration

**Wave 1 config (DEDUP_WORKER_ENABLED = false):**
```
Extract worker → chains to Score worker (skip Dedup)
dedupQueue.add() is NO-OP
extractions.canonical_model_id = null (or auto-filled with extraction_id)
dedupQueue remains empty (queued jobs not processed)
```

**Post-Wave-1 config (DEDUP_WORKER_ENABLED = true):**
```
Extract worker → chains to Dedup worker
Dedup worker → chains to Score worker (after dedup)
dedupQueue processes all new extractions
extractions.canonical_model_id = resolved value
If duplicate: extraction.review_queue_id set (for admin review, Phase 2.2)
```

## 6. Go/No-Go Decision Criteria (Before Enabling Flags)

**GO Criteria (all must pass):**
- [ ] Backfill completed with zero orphan extractions
- [ ] pricing_history has identical price records as extractions (spot-checked 30+)
- [ ] Phase 3 API continues serving responses in <100ms (p99 latency)
- [ ] Zero extraction failures in 24h post-backfill
- [ ] Canonical registry queries run in <10ms (p99) with indexed lookups
- [ ] Zero false positives in dedup testing (> manual sample of 100 models)
- [ ] All rollback tests pass
- [ ] On-call engineer confirms readiness

**NO-GO Criteria (any triggers rollback):**
- [ ] Orphan extractions detected (>0.1% of total)
- [ ] Pricing history diverges from extractions (any row missing)
- [ ] Phase 3 API latency increases >20%
- [ ] Any extraction failures post-backfill
- [ ] Dedup produces >1 false positive per 1000 models tested
- [ ] Any data corruption detected
- [ ] Integration tests fail
