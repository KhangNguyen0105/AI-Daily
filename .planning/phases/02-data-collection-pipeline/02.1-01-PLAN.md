**Class interface:**
```
class CanonicalRegistry {
  async register(
    providerName: string,
    providerModelId: string,
    extraction: ExtractionResult,
    sourceTier: SourceTier,
  ): Promise<string> // returns canonical_id
  
  async resolve(
    providerName: string,
    extractedName: string,
    providerModelId?: string,
  ): Promise<{canonical_id: string, canonical_name: string} | null>
  
  async addAlias(
    canonical_id: string,
    alias: string,
  ): Promise<void>
  
  async recordRename(
    canonical_id: string,
    old_name: string,
    new_name: string,
    reason: string,
  ): Promise<void>
  
  async getLineage(canonical_id: string): Promise<Array<{...}>>
  
  async markDeprecated(canonical_id: string, replacedBy?: string): Promise<void>
}
```

**Matching priority (per D-04):**
1. Exact provider model ID + provider match (highest confidence)
2. Canonical registry alias exact match
3. Normalized name match (case-insensitive, trim, remove punctuation)
4. Family match + context window match (e.g., all gpt-4-turbo with 128k context)
5. Never auto-merge below 85% match confidence

**Register flow:**
- Check if extraction already mapped to canonical model (via provider ID or alias)
- If yes: return existing canonical_id, update last_seen, record audit event
- If no: create new canonical model record with status="active", record created event
- Store provider model ID in api_model_ids array
- Link extraction to canonical model in extractions table

**Resolve flow:**
- Try provider model ID lookup first (highest priority per D-04)
- Then alias lookup
- Then normalized name lookup
- Return canonical_id + canonical_name or null

**AddAlias flow:**
- Append to aliases[] array
- Record audit event "aliased"
- Update last_updated timestamp

**RecordRename flow:**
- Append old_name to aliases[] if not already present
- Update canonical_name to new_name
- Record audit event "renamed" with reason
- Update lineage JSONB with timestamp and old->new mapping

**Per D-04:** Provider model IDs are the strongest identity signal. Always prefer them over fuzzy matching.

Create src/lib/deduplication.ts with deduplication logic:

**Exports:**
```
export interface DuplicateDetectionScore {
  modelName: string;
  candidates: Array<{
    canonical_id: string;
    canonical_name: string;
    matchScore: number; // 0-100
    signals: {
      providerIdMatch?: number;
      aliasMatch?: number;
      normalizedNameMatch?: number;
      familyAndContextMatch?: number;
      pricingSimilarity?: number;
    };
    confidence: 'high' | 'medium' | 'low';
  }>;
  recommendation: 'auto_link' | 'review_queue' | 'new_model';
}

export async function detectDuplicates(
  extraction: ExtractionResult,
  providerName: string,
  canonicalRegistry: CanonicalRegistry,
): Promise<DuplicateDetectionScore>

export async function calculateMatchScore(
  extraction: ExtractionResult,
  candidate: any, // canonical model from DB
): Promise<{score: number, signals: {...}}>

export async function generateReviewQueue(
  extraction: ExtractionResult,
  candidates: DuplicateDetectionScore,
): Promise<ReviewQueueItem>
```

**Matching algorithm (weighted signals):**
1. Provider model ID exact match: 100% if matched, 0 if not (highest weight, binary)
2. Canonical alias exact match: 90% (very high confidence)
3. Normalized name exact match: 75% (good confidence)
4. Family + context window match: 60% (medium confidence — both family and context must match)
5. Pricing similarity: 40% if within 5% input+output price range (weak signal, use as tiebreaker)

**Final score calculation:**
- High confidence (>85%): auto-link to canonical model
- Medium confidence (55-84%): route to review queue with explanation
- Low confidence (<55%): treat as new model

**Review queue format (for admin dashboard):**
```
{
  extraction_id: int,
  extracted_model_name: string,
  candidates: [
    {
      canonical_id: string,
      canonical_name: string,
      match_score: number,
      match_reason: string, // "Provider ID match", "Alias match", etc.
      recommendation_strength: 'strong' | 'moderate' | 'weak'
    }
  ],
  status: 'pending',
  created_at: timestamp
}
```

**Conflict handling (per D-04):**
- If fuzzy match confidence < 55% and no provider ID match: create new canonical model
- If multiple candidates >85%: escalate to review (logic error — shouldn't happen with provider IDs)
- If price change detected (>20% diff from previous): add "price_change_flag" to review item
- Never auto-delete or merge without evidence

**Workflow:**
1. For each new extraction, call detectDuplicates()
2. If recommendation == 'auto_link': immediately link to canonical_model_id
3. If recommendation == 'review_queue': create review queue item (for Phase 2.2 admin interface)
4. If recommendation == 'new_model': register as new canonical model via CanonicalRegistry

Per D-04: Fuzzy matching is candidates-only, never auto-merges below 85%. Provider model IDs trump all fuzzy matching.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Extraction → Canonical Registry | Untrusted extracted model name crosses boundary; must validate before storing |
| Provider API → Canonical Registry | Provider model IDs are strongest signal; treat as trusted identity source |
| Dedup worker → Public UI | Review queue items are internal; must never leak to public UI |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | Canonical model registration | mitigate | CanonicalRegistry.register validates extraction completeness and runs two-pass verification before storing (re-uses existing verification module) |
| T-02-02 | Tampering | Provider model ID spoofing | mitigate | Store raw provider model ID + (provider, extraction_timestamp); validate against live API list in verification pass |
| T-02-03 | Information Disclosure | Historical pricing log | mitigate | pricing_history is append-only, never overwrites; admin-only access (Phase 2.2); queryable for audit trail |
| T-02-04 | Denial of Service | Dedup worker performance | mitigate | Dedup matching uses indexed provider_id + normalized_name lookups; fuzzy matching only triggers on no provider ID; time-bound to 30 sec/extraction max |
| T-02-05 | Repudiation | Model rename tracking | mitigate | model_status_audit immutable log records all renames with timestamp + triggered_by; lineage JSONB preserves full rename chain |

</threat_model>
